## 4. Внешние зависимости и интеграции

### 4.1 Реестр внешних зависимостей

| **Система** | **Назначение** | **Протокол** | **Синхр./Async** | **Устойчивость при сбое** |
|-------------|----------------|--------------|------------------|---------------------------|
| 1С Склад | Проверить, зарезервировать и списать запчасти | REST :443 | Синхронно при проверке (ждём до 3 сек). Асинхронно при списании | При недоступности — используем кэш (5 минут), ставим отметку "запчасти не подтверждены". Повторяем позже по заданию |
| Keycloak/ADFS | Вход пользователя, проверка JWT-токенов | HTTPS :443 | Синхронно при входе. JWT-ключи кэшируем | JWT-ключи в кэше Redis (1 час). Сам токен живёт 15 минут — система работает без доступа к IdP |
| LDAP/AD | Загрузка пользователей и их ролей | LDAP :636 TLS | Фоновое задание | При сбое — используем закэшированные сессии |
| ERP-система | Отправить данные о затратах | REST :443 | Асинхронно | При ошибке — ставим в очередь ошибок (DLQ). Повторяем 3 раза с паузами. Если не помогло — шлём аларм |
| SCADA/АСУ ТП | Получить статус оборудования и срочные сигналы | OPC-UA/REST :443 | Статус — синхронно. Сигналы — асинхронные push-уведомления | При недоступности — включаем защиту, ставим флаг "SCADA недоступна" |
| SMTP/Push-сервис | Уведомления | SMTP :587 / webhook | Асинхронно | Повторяем 1 раз. Дублирующие уведомления не страшны |

### 4.2 Правила устойчивости

- Синхронные интеграции (1С checkAndReserve, SCADA) → Circuit Breaker (Resilience4j): CLOSED→OPEN при 5 ошибках за 10 сек; Half-Open через 30 сек; таймаут ≤ 3 сек.
- Async-интеграции (ERP, deductParts) → DLQ; 3 попытки с exponential backoff (30 сек, 5 мин, 30 мин); алерт при DLQ size > 0.
- IdP/LDAP → кэш JWKS в Redis TTL 1ч; при полной недоступности — JWT работают 15 мин; Refresh → мягкая деградация.

### 7.3 Правило управления секретами

- Нулевой секрет в репозитории: git-secrets + trufflehog в pre-commit hook и CI (блокирует MR).
- Секреты поступают в контейнер только через Vault Agent Sidecar; фильтруются из логов.
- Ротация: Dynamic Secrets PostgreSQL — ежемесячно автоматически. Статические (client_secret) — ежемесячная ручная ротация с записью в Change Log.

### 8.3 Порядок диагностики инцидента

Стандартный 8-шаговый алгоритм RCA из ПР №9 (разд.9.3) применяется в полном объёме. Дополнения для инфраструктурных инцидентов:

- Недоступность App Server: Grafana → «App Zone» → CPU/Memory → kubectl rollout status → Loki {level=ERROR} последние 100 строк.
- Медленная БД: Jaeger → span duration > 1 сек с тегом db.\* → операция → pg_stat_statements → EXPLAIN ANALYZE → индекс или рефакторинг запроса.
- DLQ растёт: Grafana toir_erp_dlq_size → Loki ERP_EXPORT_RETRY → errorMsg → проверить ERP endpoint → manual replay через /admin/retry-dlq.
- Сбой cron ППР: Loki {component=PprScheduler} → нет PPR_REQUEST_CREATED за день → kubectl CronJob status → ручной запуск через API.

## 9. Безопасность развёртывания и управление обновлениями

### 9.1 Сетевые ограничения и минимальные права

| **Принцип** | **Применение** | **Реализация** |
|-------------|----------------|-----------------|
| Zero Trust (сетевые сегменты) | Трафик между зонами — явный allowlist; всё остальное DENY | Firewall DENY ALL по умолчанию; allowlist по IP-блокам зон (ПР №4 разд.2.2) |
| Минимальные права БД | toir_app: SELECT/INSERT/UPDATE/DELETE только в схеме toir; toir_readonly: только SELECT; audit_writer: только INSERT в audit_log | GRANT/REVOKE в Flyway-миграциях; проверка прав в CI (psql \dp) |
| Минимальные права ОС | App Server — непривилегированный пользователь (uid 1000); нет root | Dockerfile: USER 1000; Kubernetes: securityContext.runAsNonRoot=true |
| Минимальные права контейнера | read-only filesystem; нет capabilities; нет hostNetwork | Kubernetes: readOnlyRootFilesystem=true; allowPrivilegeEscalation=false |
| Сканирование образов | Каждый build: Trivy Critical/High → FAIL | CI-шаг 'trivy-scan': trivy image --severity CRITICAL,HIGH |
| Обновления зависимостей | Dependabot/Snyk еженедельно; Critical → 3 раб. дня; High → следующий спринт | GitHub Dependabot + Snyk в CI; репорт в Jira автоматически |
| Обновления ОС | Автоматические security patches; перезагрузка в окно 02:00–04:00 воскресенье | Ansible unattended-upgrades + maintenance window в мониторинге |

### 9.2 Журналирование административных действий

- SSH-сессии через Bastion записываются (session recording); хранение 1 год; доступ только ИБ.
- Операции через Vault (просмотр секретов, ротация) логируются в Vault audit log; хранение 3 года.
- Прямые операции с БД (psql через Bastion) логируются pgaudit; все DDL и SELECT на toir/audit — в лог.
- Изменения инфраструктуры (Terraform apply, Ansible) — git MR + CHANGELOG.md.

### 9.3 Порядок выпуска версий, отката и проверки совместимости

| **Этап** | **Действие** | **Критерий / откат** |
|----------|--------------|----------------------|
| 1. Code Freeze и тегирование | git-тег release-vX.Y.Z; CHANGELOG; обновление OpenAPI ART-12 | Все MR смержены; pipeline зелёный; swagger-validate OK |
| 2. Сборка и сканирование | Docker build; Trivy; Unit + Integration тесты | 0 Critical уязвимостей; все тесты зелёные |
| 3. Деплой в staging | helm upgrade; smoke-тесты TC-P01, TC-P08; нагрузочный тест 30 мин | P95 ≤ 2 сек при 100 VU; 0 ошибок P1/P2 |
| 4. Деплой в prod (rolling) | kubectl rollout: одна реплика за раз; healthcheck перед переключением трафика | Readiness probe OK; error rate не растёт > 0.1% за 5 мин |
| 5. Мониторинг 30 мин | Grafana: error_rate, P95, DB connections | При error_rate > 1% → автоматический rollback (kubectl rollout undo) |
| 6. Откат | kubectl rollout undo; Flyway undo (только обратимо-совместимые миграции) | Откат ≤ 5 мин; данные не теряются (forward-only миграции) |
| 7. Contract-тесты совместимости | Pact consumer-driven contract tests UI↔API; при breaking change → /api/v2 | Все Pact-тесты зелёные; breaking changes запрещены без версионирования |

### 9.4 Стратегия Flyway-миграций

- Forward-only: нет DROP без deprecated-периода ≥ 1 релиз. Удаление колонки — 4-шаговый процесс (deprecated → новая колонка → migrate → DROP).
- Нумерация строго по порядку: V001__init.sql, V002__add_version.sql. Конфликт номеров блокирует сборку в CI.
- Миграции тестируются на чистой БД (docker run postgres:15) и на копии staging-БД еженедельно.
