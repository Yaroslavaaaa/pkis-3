## 1. Реестр поведенческих требований

| **Требование (поведенческое)** | **Критерий проверки** |
|--------------------------------|------------------------|
| При регистрации заявки система должна автоматически вычислить приоритет P1..P4 до того, как заявка станет видна диспетчеру | Заявка сохраняется с priority≠NULL; время вычисления ≤100 мс |
| При назначении наряда система должна параллельно выполнить резервирование ЗЧ в 1С и выбор исполнителя; ни одно из этих действий не должно блокировать другое | Оба результата получены до создания WorkOrder; таймаут ≤5 сек |
| При недоступности 1С (Circuit Breaker OPEN) наряд должен создаваться с флагом parts_unconfirmed=true, а не блокироваться | Наряд создан; флаг=true; retry-job запущен; диспетчер уведомлён |
| Наряд нельзя закрыть без хотя бы одного фото и при условии endTime > startTime | Попытка без фото → 422 BR-01; попытка с endTime≤startTime → 422 BR-02 |
| Заявку нельзя закрыть, пока не все связанные наряды перешли в статус ПРИНЯТ | Попытка close() при WO≠ACCEPTED → 422 BR-04; при всех ACCEPTED → CLOSED автоматически |
| При возврате наряда система должна автоматически перевести заявку из ОЖИДАЕТ_ПРИЁМКИ обратно в В_РАБОТЕ и уведомить исполнителя с текстом замечания | WO→RETURNED; RR→IN_PROGRESS (авто); уведомление с returnComment |
| При закрытии заявки система должна атомарно обновить WorkOrder, RepairRequest и EquipmentHistory в рамках одной транзакции | При сбое в середине TX-4 ни один из трёх объектов не должен быть обновлён частично |
| Экспорт затрат в ERP должен быть идемпотентным: повторная отправка с тем же workOrderId не должна создавать дублирующую проводку | При повторном вызове exportCosts(workOrderId) ERP отвечает OK без создания дубля; idempotency_key=workOrderId |
| При генерации задания ППР система должна запустить её за N дней до nextDueDate и не позднее | cron выполняется ежедневно; при nextDueDate ≤ today+N создаётся RepairRequest типа SCHEDULED |
| Каждое изменение состояния заявки или наряда должно быть зафиксировано в журнале аудита до возврата ответа пользователю | AuditLog содержит запись с тем же timestamp, что и изменение статуса; отсутствие записи = баг |
| Все ключевые операции должны передавать trace_id сквозь все слои приложения и во внешние системы | trace_id присутствует в логах каждого компонента для одного запроса; найти запрос в Jaeger по trace_id |
| При P1-заявке система должна отправить уведомление диспетчеру немедленно (≤30 сек от момента создания), не дожидаясь ручного обновления | Уведомление получено ≤30 сек; при P1 уведомляется также руководитель |

## 3. Набор диаграмм для каждого сценария

### 3.1 Матрица «сценарий → диаграмма → вопрос, на который отвечает»

| **Сцен.** | **Тип диаграммы** | **Вопрос, на который отвечает** | **Источник истины для** |
|-----------|-------------------|--------------------------------|-------------------------|
| СЦ-1 | Sequence (диаграмма взаимодействия) | Какие объекты участвуют, в каком порядке, с какими данными? | Порядок вызовов, контракты сообщений |
| СЦ-1 | Activity (диаграмма деятельности) | Кто отвечает за каждый шаг, где ветвления, где ручной/авто? | Дорожки ответственности |
| СЦ-1 | State Machine (WO) | Из какого состояния разрешён переход, какие guard-условия? | Статусные переходы (источник истины) |
| СЦ-2 | Sequence (диаграмма взаимодействия) | Порядок вызовов при accept; что входит в TX-4? | Порядок вызовов |
| СЦ-2 | Activity (диаграмма деятельности) | Роль начальника участка, параллельные post-close действия | Дорожки ответственности |
| СЦ-2 | State Machine (RR + WO + Defect) | Что допустимо после ПРИНЯТ? Что меняется в Defect? | Статусные переходы |

### 3.2 Правило «источник истины»

- Диаграммы состояний — источник истины для допустимых статусных переходов. При противоречии между Sequence и State Machine — State Machine побеждает.
- Контракты сообщений — источник истины для API-контрактов (входные/выходные данные, коды ошибок). При противоречии с Activity — ART-34 побеждает.
- Диаграммы деятельности — источник истины для распределения ответственности между ролями. При противоречии с Sequence — Activity побеждает.
- REQ-F/NF — источник истины для критериев приёмки. При противоречии с диаграммами — требования побеждают, диаграммы обновляются.

### 5.2 СЦ-2 — Параллельные post-close действия (детализация)

После фиксации TX-4 EventBus публикует RequestClosedEvent. Три независимых обработчика выполняются параллельно — ни один не блокирует другой, ни один не может откатить TX-4:

| **Обработчик** | **Действие** | **Идемпотентность** | **Поведение при сбое** | **Аудит-событие** |
|----------------|--------------|---------------------|------------------------|--------------------|
| ErpCostExportHandler | Экспортировать затраты в ERP (MSG-12) | Да — idempotency_key = workOrderId | 3 retry с backoff; затем DLQ; алерт P2 | ERP_EXPORT_SENT |
| DefectEliminateHandler | Перевести Defect→ELIMINATED (авто) | Да — ELIMINATED терминальное, повтор → no-op | Если Defect не найден — warning log; не критично | DEFECT_ELIMINATED |
| InitiatorNotifyHandler | Уведомить инициатора заявки | Нет (push уведомления допускают дубль) | При сбое — 1 retry; дубль уведомления допустим | NOTIFICATION_SENT |

## 6. Сопоставление с диаграммами состояний

### 6.1 Проверка: каждый переход в Sequence покрыт State Machine

| **Сцен.** | **Переход в Sequence** | **Ожидаемый переход в SM** | **Совпадение?** |
|-----------|------------------------|----------------------------|-----------------|
| СЦ-1 | WO: IN_PROGRESS → COMPLETED | IN_PROGRESS → COMPLETED по событию COMPLETE | Да |
| СЦ-1 | RR: IN_PROGRESS → WAITING_FOR_ACCEPTANCE | IN_PROGRESS → WAITING по событию ALL_WO_COMPLETED | Да |
| СЦ-1 | WO: COMPLETED → COMPLETED (идемпотентный повтор) | Повторный COMPLETE при status=COMPLETED → 409 | Да |
| СЦ-2 | WO: COMPLETED → ACCEPTED | COMPLETED → ACCEPTED по событию ACCEPT | Да |
| СЦ-2 | RR: WAITING → CLOSED | WAITING → CLOSED по событию CLOSE (авто) | Да |
| СЦ-2 | Defect: IN_REPAIR → ELIMINATED | IN_REPAIR → ELIMINATED по событию ELIMINATE (авто) | Да |
| СЦ-1* | TX-3 откат → статусы не меняются (WO остаётся IN_PROGRESS) | Нет перехода при откате = состояние не меняется | Да — SM не изменяется при откате |
| СЦ-2* | Параллельный accept: второй HEAD получает 409 после SELECT FOR UPDATE | ACCEPTED → ACCEPTED: переход запрещён (терминальное) | Да |

### 6.2 Атомарность изменения состояния

| **Переход состояния** | **Транзакция** | **Что входит в TX** | **Гарантия атомарности** |
|-----------------------|----------------|---------------------|--------------------------|
| WO: IN_PROGRESS → COMPLETED + RR: IN_PROGRESS → WAITING | Закрытие наряда | UPDATE work_orders + UPDATE repair_requests | Spring @Transactional; PgBouncer в transaction mode; при откате оба UPDATE отменяются |
| WO: COMPLETED → ACCEPTED + RR: WAITING → CLOSED + EquipmentHistory INSERT | Приемка работ | UPDATE work_orders + UPDATE repair_requests + INSERT equipment_history | Spring @Transactional; SELECT FOR UPDATE на WO перед обновлением |
| RR: [*] → NEW (ППР) + проверка идемпотентности | Создание заявки | SELECT COUNT + INSERT repair_requests | Уровень изоляции SERIALIZABL предотвращает дублирование при параллельных cron-запусках |
| Defect: IN_REPAIR → ELIMINATED | Устранение деффекта | UPDATE defects.status = ELIMINATED WHERE status = IN_REPAIR | Идемпотентность: WHERE status = IN_REPAIR; повторный вызов = 0 строк, не ошибка |

## 7. Обработка ошибок, идемпотентность и компенсация

### 7.1 Идемпотентность операций

| **Операция** | **Почему требует идемпотентности** | **Механизм идемпотентности** | **Что происходит при повторном вызове** |
|--------------|-----------------------------------|------------------------------|----------------------------------------|
| PATCH /complete (закрытие наряда) | Мобильный клиент может повторить запрос при плохой сети | Guard в WO.complete(): status≠IN_PROGRESS → 409 CONFLICT | Повторный вызов при status=COMPLETED → 409; данные не изменяются |
| POST /accept (приёмка наряда) | Медленная сеть; двойной клик | SELECT FOR UPDATE + guard status=COMPLETED; при status=ACCEPTED → 409 | Второй HEAD получает 409; первый получил 200 |
| exportCosts(workOrderId) → ERP | retry при недоступности ERP (DLQ) | idempotency_key = workOrderId отправляется в ERP; ERP дедуплицирует | ERP возвращает OK без создания дублирующей проводки |
| deductParts(workOrderId) → 1С | retry-job при недоступности 1С | idempotency_key = workOrderId в заголовке HTTP-запроса к 1С | 1С возвращает OK; повторного списания не происходит |
| triggerPprCheck() → cron | cron может запуститься дважды при сбое планировщика | SELECT COUNT + INSERT в SERIALIZABLE TX; при COUNT>0 → skip | Вторая попытка: COUNT=1 → PPR_ALREADY_GENERATED в логе; заявка не создаётся |
| DefectEliminateHandler (async) | RequestClosedEvent может быть доставлен дважды (at-least-once) | UPDATE ... WHERE status = IN_REPAIR (не ELIMINATED) | Повторный вызов: 0 строк обновлено; no-op; success |

### 7.2 Компенсационные действия

| **Сценарий частичного выполнения** | **Что уже сделано** | **Компенсационное действие** | **Кто выполняет** |
|-------------------------------------|---------------------|------------------------------|--------------------|
| Закрытие наряда (откат при сохранении наряда) | WO.complete() выполнен в памяти, данные НЕ в БД; SPA ещё не вызван | Откат автоматический (Spring TX); компенсация не нужна | Автоматически |
| Закрытие наряда (транзакция зафиксирована, но ЗЧ НЕ списаны в 1С из-за недоступности 1С) | WO=COMPLETED, RR=WAITING в БД; ЗЧ НЕ списаны в 1С | Фоновый retry-job: SELECT WO WHERE deduction_pending=true → повтор deductParts с idempotency_key | PprScheduler / RetryJob |
| Закрытие наряда (транзакция зафиксирована, но списание запчастей не вызвано из-за недоступности 1С) | WO.accept() выполнен в памяти; БД не обновлена | Откат автоматический; SELECT FOR UPDATE освобождён; EventBus не вызывается | Автоматически |
| Приёмка работ (транзакция зафиксирована, но ERP НЕ получила данные) | WO=ПРИНЯТ, RR=ЗАКРЫТА в БД; ERP НЕ получила данные | 3 retry с backoff; при исчерпании → DLQ + алерт P2; ручной повтор или CSV-выгрузка | ErpCostExportHandler + администратор |
| Назначение наряда (запчасти зарезервированы, но WorkOrder НЕ создан в БД) | reservationId создан в 1С; WorkOrder НЕ создан в БД | Компенсация: при откате транзакции отменить резерв в 1С | PprService |

### 7.3 Поведение при нарушении прав доступа

| **Операция** | **Нарушение** | **HTTP-ответ** | **Аудит и мониторинг** |
|--------------|---------------|----------------|------------------------|
| PATCH /complete от пользователя ≠ assignedTo | роль ENGINEER, но не назначенный исполнитель | 403 FORBIDDEN {required: assignedTo, actual: userId} | WARN: unauthorized_complete_attempt; counter toir_auth_violations_total{endpoint} |
| POST /accept от ENGINEER (не HEAD_OF_UNIT) | недостаточно прав на приёмку | 403 FORBIDDEN {required_role: HEAD_OF_UNIT} | WARN: unauthorized_accept_attempt; алерт при > 5/мин — возможна атака |
| Прямой SQL UPDATE в audit_log | попытка изменения аудит-записи | PostgreSQL error permission denied (REVOKE UPDATE) | ERROR в PostgreSQL log; алерт ИБ-специалисту немедленно |
| Попытка CANCEL из IN_PROGRESS без approvedByHead | бизнес-правило нарушено | 422 BUSINESS_RULE_VIOLATION {rule: approvedByHead required} | WARN: business_rule_cancel_without_approval |

### 9.3 Алгоритм поиска первопричины по сценарию

#### Шаблон диагностики (применимый к любому сценарию)

| **Действие** | **Инструмент / запрос** |
|--------------|------------------------|
| Получить trace_id из жалобы пользователя или из заголовка X-Trace-Id ответа | Пользователь сообщает trace_id; или: SELECT trace_id FROM audit_log WHERE entity_id = :requestId ORDER BY ts DESC LIMIT 1 |
| Найти все span для этого trace_id в Jaeger | Jaeger UI → поиск по trace_id → дерево span с временными метками |
| Определить span с наибольшим duration или со статусом ERROR | В Jaeger: отсортировать span по duration; или фильтр status=ERROR |
| По span.service открыть логи компонента в Loki за ±1 мин от span.startTime | Loki: {service=<span.service>} |= <trace_id> |
| Если ошибка в домене (domain): найти ruleId в WARN-логе BUSINESS_RULE_VIOLATION | AppLog: {level=WARN} |= 'BUSINESS_RULE_VIOLATION' |= <trace_id> |
