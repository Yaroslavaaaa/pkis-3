## 1. Выбор сущностей для моделирования состояний

### 1.1 Выбранные сущности и обоснование

| **Сущность** | **Текущие статусы** | **Риски без формальной модели** |
|--------------|---------------------|----------------------------------|
| RepairRequest (Заявка на ремонт) | НОВАЯ, В_РАБОТЕ, ОЖИДАЕТ_ПРИЁМКИ, ЗАКРЫТА, ОТЛОЖЕНА, ОТМЕНЕНА | Заявка может закрыться без приёмки; диспетчер может отменить заявку в работе без фиксации; два наряда на одну заявку в состоянии НОВАЯ |
| WorkOrder (Наряд на выполнение работ) | НАЗНАЧЕН, В_РАБОТЕ, ВЫПОЛНЕН, ПРИНЯТ, ВОЗВРАЩЁН, ОТМЕНЁН | Исполнитель закрывает наряд без фото (BR-01); наряд принимается без перехода заявки; два пользователя принимают один наряд параллельно |
| Defect (Дефект) | REGISTERED, CONFIRMED, ELIMINATED | Дефект помечается устранённым до закрытия наряда; дефект не привязан к заявке при устранении; повторное открытие дефекта на том же узле без нового обследования |

### 1.2 Состояния, которые нужно уточнить

- RepairRequest.ОТЛОЖЕНА — статус упомянут, но переходы не определены. Уточняем: кто откладывает, на какой срок, из каких состояний.
- WorkOrder.ВОЗВРАЩЁН — описан как альтернативный поток для Принять выполненные работы, но из него возможен возврат только в В_РАБОТЕ. Уточняем: возможна ли повторная приёмка из ВОЗВРАЩЁН без нового complete().

## 2. Полный набор состояний

### 2.1 RepairRequest — состояния

| **Состояние (код)** | **Смысл** | **Терминальное** | **Допускает возврат назад** |
|---------------------|-----------|------------------|-----------------------------|
| НОВАЯ (NEW) | Заявка зарегистрирована; ещё не назначен ни один наряд; ожидает действия диспетчера | Нет | Нет |
| ОТЛОЖЕНА (DEFERRED) | Диспетчер временно отложил обработку заявки (нет исполнителя / нет ЗЧ); ожидает повторного рассмотрения | Нет | Нет |
| В_РАБОТЕ (IN_PROGRESS) | Хотя бы один наряд назначен и находится в работе; заявка активно исполняется | Нет | Нет |
| ОЖИДАЕТ_ПРИЁМКИ (WAITING_FOR_ACCEPTANCE) | Все наряды переведены в ВЫПОЛНЕН; ждёт подтверждения начальника участка | Нет | Да |
| ЗАКРЫТА (CLOSED) | Все наряды приняты; начальник участка подтвердил; история оборудования обновлена | Да | Нет |
| ОТМЕНЕНА (CANCELLED) | Заявка аннулирована; ни один наряд не должен быть в активном статусе | Да | Нет |

### 2.2 WorkOrder — состояния

| **Состояние (код)** | **Смысл** | **Терминальное** | **Допускает возврат назад** |
|---------------------|-----------|------------------|-----------------------------|
| НАЗНАЧЕН (ASSIGNED) | Наряд сформирован и передан исполнителю; работы не начались | Нет | Нет |
| В_РАБОТЕ (IN_PROGRESS) | Исполнитель нажал «Начать работы»; зафиксирован startTime | Нет | Нет |
| ВЫПОЛНЕН (COMPLETED) | Исполнитель закрыл наряд: фото, ЗЧ, endTime зафиксированы; ожидает приёмки | Нет | Нет |
| ВОЗВРАЩЁН (RETURNED) | Начальник участка отклонил результат с комментарием; требуется повторное выполнение | Нет | Да |
| ПРИНЯТ (ACCEPTED) | Начальник участка подтвердил качество работ; наряд завершён | Да | Нет |
| ОТМЕНЁН (CANCELLED) | Наряд аннулирован (до начала работ или в НАЗНАЧЕН); зарезервированные ЗЧ освобождены | Да | Нет |

### 2.3 Defect — состояния

| **Состояние (код)** | **Смысл** | **Терминальное** | **Допускает возврат назад** |
|---------------------|-----------|------------------|-----------------------------|
| REGISTERED (REGISTERED) | Дефект зафиксирован исполнителем или инженером; ещё не верифицирован | Нет | Нет |
| CONFIRMED (CONFIRMED) | Инженер-механик или начальник участка подтвердил наличие дефекта; дефект требует ремонта | Нет | Да |
| IN_REPAIR (В_РЕМОНТЕ) | Для дефекта создана заявка и хотя бы один наряд в работе | Нет | Нет |
| ELIMINATED (УСТРАНЁН) | Связанная заявка закрыта, дефект устранён; узел оборудования возвращён в норму | Да (необратимо — если узел в порядке; иначе новый дефект) | Нет |

## 3. События и источники событий

### 3.1 События RepairRequest

| **Событие (код)** | **Описание** | **Инициатор** | **Автоматическое** |
|-------------------|--------------|---------------|---------------------|
| CREATE | Заявка создана с описанием дефекта, узлом и приоритетом | Инженер, Диспетчер | Нет |
| DEFER | Диспетчер откладывает заявку (причина обязательна) | Диспетчер | Нет |
| REACTIVATE | Диспетчер возвращает отложенную заявку в очередь | Диспетчер | Нет |
| ALL_WO_COMPLETED | Все наряды по заявке перешли в ВЫПОЛНЕН | Система | Да |
| RETURN_TO_WORK | Один из нарядов возвращён → заявка возвращается В_РАБОТЕ | Система | Да |
| CLOSE | Все наряды приняты; начальник участка подтверждает закрытие | Система | Да |
| CANCEL | Заявка аннулирована; причина обязательна; все активные WO должны быть ОТМЕНЕНЫ | Диспетчер | Нет |

### 3.2 События WorkOrder

| **Событие (код)** | **Описание** | **Инициатор** | **Автоматическое** |
|-------------------|--------------|---------------|---------------------|
| CREATE | Наряд создан диспетчером и назначен исполнителю | Диспетчер | Нет |
| START | Исполнитель начал работы; фиксируется startTime | Инженер / Подрядчик | Нет |
| COMPLETE | Исполнитель закрыл наряд с фото и ЗЧ | Инженер / Подрядчик | Нет |
| ACCEPT | Начальник участка принял результат | Начальник участка | Нет |
| RETURN | Начальник вернул наряд с комментарием | Начальник участка | Нет |
| RESUME | Исполнитель ознакомился с замечанием и возобновил работы | Инженер / Подрядчик | Нет |
| CANCEL | Наряд отменён до начала работ; ЗЧ освобождены в 1С | Диспетчер (только из ASSIGNED) | Нет |

### 3.3 События Defect

| **Событие (код)** | **Описание** | **Инициатор** | **Автоматическое** |
|-------------------|--------------|---------------|---------------------|
| REGISTER | Дефект зафиксирован на узле оборудования | Инженер / Диспетчер / SCADA | Нет |
| CONFIRM | Инженер-механик или нач. участка верифицировал дефект | Инженер, Начальник участка | Нет |
| LINK_TO_REQUEST | Для дефекта создана заявка, дефект переходит IN_REPAIR | Система | Да |
| ELIMINATE | Связанная заявка закрыта → дефект помечается устранённым | Система | Да |

## 5. Инварианты и обязательные данные по состояниям

### 5.1 Инварианты RepairRequest

| **Состояние** | **Обязательные данные** | **Запрещённые данные** | **Уровень контроля** |
|---------------|-------------------------|------------------------|----------------------|
| NEW | id, requestNumber, createdAt, createdBy, equipmentNodeId, defectType, description(≥10), priority, status=NEW | closedAt, cancelledAt, cancelReason | База данных (NOT NULL) + Конструктор домена |
| DEFERRED | Всё из NEW + deferReason, deferredAt, deferredBy | closedAt, cancelledAt | Метод defer() + проверка в БД (причина не пустая) |
| IN_PROGRESS | Всё из NEW + связан хотя бы один WorkOrder (activeWO ≥ 1) | closedAt, cancelledAt | Метод setInProgress() + проверка на уровне приложения |
| WAITING_FOR_ACCEPTANCE | Всё из IN_PROGRESS + все WO в статусе COMPLETED | closedAt, cancelledAt | Проверка в домене (все наряды выполнены) |
| CLOSED | Всё из WAITING + closedAt, все WO = ACCEPTED | cancelledAt; нет активных WO | Метод close() + БД (NOT NULL дата закрытия) |
| CANCELLED | Всё из любого предыдущего + cancelReason, cancelledAt, cancelledBy | closedAt (взаимоисключают) | Метод cancel() + проверка в БД |

### 5.2 Инварианты WorkOrder

| **Состояние** | **Обязательные данные** | **Запрещённые данные** | **Уровень контроля** |
|---------------|-------------------------|------------------------|----------------------|
| ASSIGNED | id, orderNumber, requestId, assignedTo (active user), plannedDate, status=ASSIGNED, createdAt | startTime, endTime, photoUrls, returnComment | Конструктор домена + внешний ключ в БД (пользователь активен) |
| IN_PROGRESS | Всё из ASSIGNED + startTime (NOT NULL, ≤ now()) | endTime, photoUrls, returnComment | Метод start() + БД (время начала не пустое) |
| COMPLETED | Всё из IN_PROGRESS + endTime(>startTime), photoUrls(≥1 URL), workDescription(NOT NULL) | returnComment (пусто) | Метод complete() + проверка в БД |
| RETURNED | Всё из COMPLETED + returnComment(≥20 символов), returnedBy, returnedAt | — | Метод complete() + проверка в БД |
| ACCEPTED | Всё из COMPLETED + acceptedBy, acceptedAt | returnComment (если не было возврата) | Метод accept() + БД (кто и когда принял — не пустые) |
| CANCELLED | id, requestId, cancelledAt, cancelledBy + отсутствие startTime (если до IN_PROGRESS) | endTime (если не было работ) | Метод cancel() — только из статуса НАЗНАЧЕН |

### 5.3 Дефект (Defect)

| **Статус** | **Обязательные данные** | **Запрещённые данные** | **Где проверяется** |
|------------|-------------------------|------------------------|---------------------|
| REGISTERED | id, equipmentNodeId, detectedAt, registeredBy, description, status = REGISTERED | confirmedAt, eliminatedAt, repairRequestId | Конструктор домена + БД (NOT NULL) |
| CONFIRMED | Всё из REGISTERED + confirmedAt, confirmedBy | eliminatedAt, repairRequestId | Метод confirm() + БД (confirmedAt NOT NULL при status = CONFIRMED) |
| IN_REPAIR | Всё из CONFIRMED + repairRequestId (ссылка на заявку) | eliminatedAt | Метод linkToRequest() (автоматически) + БД (repairRequestId NOT NULL при status = IN_REPAIR) |
| ELIMINATED | Всё из IN_REPAIR + eliminatedAt, repairRequest.status = CLOSED | — | Метод eliminate() (автоматически при закрытии заявки) + БД (eliminatedAt NOT NULL при status = ELIMINATED) |

### 5.4 Глобальные инварианты (нарушение недопустимо ни при каком переходе)

- **Заявка закрыта** → **все наряды приняты**. И наоборот: если все наряды приняты, заявка должна быть закрыта. Контроль: триггер в базе данных.
- **Время окончания работ** всегда **позже времени начала**. Контроль: проверка в домене + ограничение CHECK в базе данных.
- **У выполненного или принятого наряда** обязательно есть **хотя бы одно фото**. Контроль: проверка в домене + ограничение CHECK в базе данных.
- **Дефект считается устранённым** только тогда, когда **связанная заявка закрыта**. Контроль: автоматический переход ELIMINATE.
- **Закрытую или отменённую заявку** нельзя изменить — это конечные состояния. Контроль: база данных запрещает обновление статуса через RLS или триггер.

### 6.4 Сообщение об ошибке при запрещённом переходе

Если пользователь пытается инициировать переход, на который не имеет прав, система возвращает HTTP 403 FORBIDDEN с телом:

```json
{
  "error": {
    "code": "FORBIDDEN",
    "message": "Недостаточно прав для выполнения операции",
    "details": [
      {
        "field": "role",
        "issue": "required_role: DISPATCHER, actual_role: ENGINEER"
      }
    ],
    "request_id": "uuid",
    "timestamp": "ISO-8601"
  }
}
```

Если переход запрещён из-за текущего состояния объекта (а не прав), система возвращает HTTP 409 CONFLICT:

```json
{
  "error": {
    "code": "CONFLICT",
    "message": "Недопустимый переход: Заявка находится в статусе ЗАКРЫТА",
    "details": [
      {
        "field": "status",
        "currentValue": "CLOSED",
        "allowedFrom": ["NEW","DEFERRED"]
      }
    ],
    "request_id": "uuid"
  }
}
```

| IN_REPAIR | ELIMINATE | ELIMINATED | (авто) связанный RepairRequest.status == CLOSED | eliminatedAt = now() | — (авто, ждёт закрытия заявки) |
| REGISTERED | — | — | CANCEL (не предусмотрен) | — | 409: дефект нельзя отменить, только подтвердить или нет |
| CONFIRMED | — | — | CANCEL (не предусмотрен) | — | 409: дефект нельзя отменить, только перевести в ремонт |
| ELIMINATED | любое | — | ЗАПРЕЩЕНО (терминальное) | — | 409 CONFLICT |

## 8. Конфликты параллельных изменений

### 8.1 Механизм предотвращения конфликтов

| **Сценарий конфликта** | **Пример** | **Механизм** | **Поведение системы** |
|------------------------|------------|--------------|----------------------|
| Два диспетчера назначают наряды на одну заявку ОДНОВРЕМЕННО | Диспетчер A и Диспетчер B оба видят заявку в статусе NEW и создают WorkOrder | Система проверяет версию заявки. Кто сохранил первым — тот победил. Второй получает ошибку. | 409 CONFLICT: «Заявка изменена другим пользователем. Обновите страницу и повторите.» + актуальный status и version в ответе |
| Исполнитель дважды нажимает «Закрыть наряд» (double submit) | Мобильный клиент отправляет два PATCH /complete подряд из-за плохой сети | После первого закрытия статус наряда меняется. Второй запрос видит, что наряд уже не «В работе», и отклоняется. | 409 CONFLICT: «Наряд уже закрыт. Текущий статус: ВЫПОЛНЕН.» |
| Два начальника участка принимают один наряд (редкий сценарий при смене смен) | Начальник A и Начальник B одновременно открывают COMPLETED наряд и нажимают «Принять» | Система блокирует наряд на время обработки. Второй начальник ждёт, потом видит, что наряд уже принят. | 409 CONFLICT: «Наряд уже принят пользователем [headA].» |
| Диспетчер отменяет заявку, пока исполнитель закрывает наряд | CANCEL заявки из IN_PROGRESS и COMPLETE наряда одновременно | Система обрабатывает их по очереди. Если возникает зависание (deadlock), система автоматически повторяет попытку (до 3 раз). | Deadlock: система повторяет отменившую транзакцию (exponential backoff, 3 попытки). Пользователь видит 409 или 503 с retry-after. |

### 8.2 Поля версионности в БД

В таблицах repair_requests и work_orders добавляется столбец version BIGINT NOT NULL DEFAULT 0. Каждое изменение увеличивает version на 1. JPA/@Version аннотация обеспечивает автоматическую проверку. При OptimisticLockException приложение перехватывает исключение и возвращает 409 с телом, включающим currentVersion и message.

Для Defect оптимистичная блокировка не критична (переходы автоматические), но version добавляется для единообразия и возможности аудита изменений.
